# P8.1 esbuild 打包设计（Batch B · 独立长尾 1）

> 日期：2026-04-20
> 范围：`.claude/worktrees/si-search-optimization`
> 背景：R6 backlog Tier 2 · P8.1 (VSIX 从 ~10MB → ~2MB)
> 前置：Batch A 已完成（P7.3/P7.4/P7.5 + R5/R6/R7 清理）

---

## 一、目标

将扩展的运行时 JS 产物从"`tsc` 编译 + 全量 `node_modules`"打包模式切换为"esbuild 单文件 bundle + 按需保留运行时依赖"。核心收益：

- VSIX 体积约 `10MB → 2MB`（减少运行时 node_modules 副本）
- 冷启动加载时间下降（single file vs 多文件 require 链）
- 保留 `tsc` 作为类型检查器（不走 esbuild 的 TS 能力）

---

## 二、现状

`package.json` 当前：

- `main: ./out/src/extension.js`
- `compile: tsc -p ./`
- `vscode:prepublish: copy-wasm && compile`
- `.vscodeignore` 已经 ignore `src/**`、`test/**`，但仍打包完整 `node_modules`

**打包完整 node_modules 的成本**：
- `@msgpack/msgpack` ~150KB
- `shiki` 含大量语言/主题 JSON，几 MB
- `tree-sitter-*` wasm 保留，但 `src/**` 已 ignore
- `web-tree-sitter` debug 目录也已 ignore

剩余主要体积来源：`shiki` 的 JSON 资源 + 多个 node_modules 包的 `package.json`/`README` 等元数据。

---

## 三、设计决策

### 3.1 打包工具选择

**选 esbuild**，理由：
- 零配置，调用即 bundle
- 速度快（10-100x tsc）
- VS Code 官方 yo-code 模板推荐
- 与 tsc 搭配稳定（esbuild 只做 bundle，tsc 做 typecheck）

不选 webpack：配置复杂、速度慢、对此场景收益不大。

### 3.2 产物布局

```
dist/
  extension.js       （单文件 bundle，~500KB）
  extension.js.map   （sourcemap，dev only）
out/                 （tsc 产物，仅 typecheck/测试用，不发布）
```

`.vscodeignore` 追加：
```
out/**
node_modules/**
!node_modules/@vscode/ripgrep/bin/**   # 保留 rg 二进制
!node_modules/tree-sitter-c/*.wasm
!node_modules/tree-sitter-cpp/*.wasm
!node_modules/web-tree-sitter/tree-sitter.wasm
```

`package.json` 改：
```json
"main": "./dist/extension.js"
```

### 3.3 external 清单（关键）

esbuild bundle 时哪些不 inline：

| 模块 | 为什么 external | 策略 |
|---|---|---|
| `vscode` | API 由宿主注入 | `--external:vscode` |
| `@vscode/ripgrep` | 需访问二进制文件路径 | `--external:@vscode/ripgrep` |
| `tree-sitter-c` / `tree-sitter-cpp` | wasm 文件加载 | external |
| `web-tree-sitter` | 同上 | external |
| 其他（`shiki`、`@msgpack/msgpack`）| 纯 JS，可 inline | bundle 进来 |

**风险**：`shiki` inline 后单文件可能膨胀到 2-3MB（含语言定义）。如确认，再拆 `shiki` 为 external 或按需加载。

### 3.4 脚本变更

```json
"scripts": {
  "copy-wasm": "node scripts/copy-wasm.js",
  "compile": "tsc -p ./ --noEmit",           // 仅类型检查
  "bundle": "node esbuild.config.js",        // 打包产物
  "watch": "node esbuild.config.js --watch",
  "vscode:prepublish": "npm run copy-wasm && npm run compile && npm run bundle -- --production",
  "test-compile": "tsc -p ./",               // 测试仍需 out/ 目录
  "test": "npm run test-compile && node ./out/test/runTest.js"
}
```

### 3.5 esbuild.config.js

单文件 Node 脚本，不引入额外 TS 工具链：
```js
// esbuild.config.js
const esbuild = require('esbuild');
const prod = process.argv.includes('--production');
const watch = process.argv.includes('--watch');

esbuild[watch ? 'context' : 'build']({
    entryPoints: ['src/extension.ts'],
    bundle: true,
    outfile: 'dist/extension.js',
    platform: 'node',
    target: 'node20',
    format: 'cjs',
    sourcemap: !prod,
    minify: prod,
    external: [
        'vscode',
        '@vscode/ripgrep',
        'tree-sitter-c',
        'tree-sitter-cpp',
        'web-tree-sitter',
    ],
    logLevel: 'info',
}).then(ctx => watch ? ctx.watch() : undefined);
```

### 3.6 `copy-wasm.js` 保留

已有脚本把 wasm 拷贝到 out/。bundle 后 wasm 加载路径需要从 `dist` 或 node_modules 定位，需要最小 patch：
- 方案 A：`copy-wasm.js` 同时拷贝到 `dist/` 旁边
- 方案 B：运行时用 `require.resolve('tree-sitter-c/tree-sitter-c.wasm')` 查找

推荐 **方案 B**（单一真相源，无冗余拷贝）。但需验证 VSIX 打包后该 resolve 路径仍成立。

## 四、测试链路影响

测试继续走 `out/`（`test-compile` 跑 tsc 全量产物）。**测试框架与产物解耦** —— bundle 不影响测试。

---

## 五、风险与回滚

| 风险 | 概率 | 缓解 |
|---|---|---|
| esbuild 不解析某个 dynamic `require` | 低 | external 清单 + 真机测试 |
| shiki/msgpack inline 后膨胀 | 中 | 监控 dist/extension.js 体积，必要时 external |
| wasm 路径解析变化 | 中 | 方案 B + 真机 smoke test |
| sourcemap 调试问题 | 低 | dev 模式保留 sourcemap |

**回滚**：单次 commit，回滚即恢复 `main: ./out/src/extension.js`。
